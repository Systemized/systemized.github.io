Security Page

This page describes the security measures applied to the website. Because the site is a static site hosted on GitHub Pages, there is no backend database or login form to attack directly. The protections below focus on transport security, browser-side hardening, and limiting what content the browser is allowed to load.

HTTPS (TLS)

GitHub Pages automatically issues a TLS certificate through Let's Encrypt and redirects HTTP traffic to HTTPS. All traffic between the browser and the server is encrypted, which prevents passive eavesdropping and tampering by anyone on the network path.

Verifying the certificate and redirect with curl:

$ curl -I http://systemized.github.io

HTTP/1.1 301 Moved Permanently
Location: https://systemized.github.io/
...

$ curl -I https://systemized.github.io

HTTP/2 200
strict-transport-security: max-age=31556952
server: GitHub.com

The strict-transport-security header (HSTS) tells the browser to only contact the site over HTTPS for the next year, even if a user types the URL without https://.

Content Security Policy

The Content Security Policy (CSP) is set through a meta tag in every page: 

<meta http-equiv="Content-Security-Policy" content="default-src 'self';">

With default-src 'self', the browser will only load scripts, styles, images, and other resources from this same origin. Any attempt to inject an inline script or load a script from an external domain is blocked. This is the main defense against XSS for a static site that has no server-side input handling.

Other Response Headers

Several additional headers are set to harden how the browser treats the page:

  • X-Frame-Options: DENY — the page cannot be embedded in an iframe, which prevents clickjacking.
  • X-Content-Type-Options: nosniff — the browser will not try to guess the type of a response, which prevents some script-injection tricks.
  • Referrer-Policy: no-referrer — no Referer header is sent when the user clicks an external link, so other sites do not learn where the user came from.
  • Permissions-Policy — disables access to the geolocation, microphone, and camera APIs, since the site does not need them.

Input Sanitization

The site does not currently take user input or write to a database, so there is no SQL injection surface. If a form is added in the future, all values written into the page should be passed through a library such as DOMPurify before being inserted into the DOM, and any database query should use parameterized statements rather than string concatenation.

Together, HTTPS, HSTS, the CSP, and the additional response headers cover the common browser-level attacks that apply to a static site: traffic interception, cross-site scripting, clickjacking, and MIME confusion.
